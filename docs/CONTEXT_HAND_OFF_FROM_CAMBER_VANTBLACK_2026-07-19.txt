PART THREE: THE METHOD YOU MUST KEEP (it is the whole point)

The program's method is ENFORCE-THEN-BUILD, also called discover-by-building. Keep it, because it is what kept the whole program honest.

  - You build something real (a konnectome region, interface, or messenger).

  - When you hit a wall - something PrologAI or Causalontology cannot express - you do NOT paper over it. You WRITE IT DOWN, precisely, in konnectome's own Requirements Ledger (konnectome_ledger.txt - it does not exist yet; you create it, exactly as PrologAI had PrologAI_Requirements_Ledger_v1.txt).

  - A load-bearing wall becomes a new construct; the next region rests on it rather than re-inventing it. The findings ARE the product.

  - HONEST NON-CLOSURE IS ALLOWED AND VALUED. A construct that "did not need building," or a wall shown to be unnecessary, is as real a result as one you built for. Record it as such. (The prior program's proudest moment was recording that a delivered construct "did not bite" rather than spinning it as a win. Hold that standard.)

PART FIVE: THE RULES YOU MUST NOT BREAK

  - CAUSALONTOLOGY IS FROZEN UNTIL KONNECTOME EARNS A CHANGE. Do not amend the standard from the armchair. A data-structure gap is only real when a REAL konnectome build hits it AND it passes the consumer-versus-ontology test: the thing must need to be SHARED across agents or repositories as a signed, evidence-graded, content-addressed record. If it can live inside one mind's PrologAI runtime, it stays a PrologAI pack; it does NOT go into the ontology. Route genuine findings to the held 4.0.0 and to konnectome_ledger.txt; never patch the standard inside a konnectome build.

  - NAMING: whole words, pack-qualified, snake_case. No terse prefixes ever (no wm_, gd_, ai_, and so on). Pack names are whole English words, never abbreviations or un-underscored concatenations. External standard names (JSON, BCP 47, SWI-Prolog, Ed25519) keep their real names.

  - ENGLISH-READABLE CODE: every line of code in every .pl file carries one plain-English comment immediately above it.

  - SPARC DOCUMENTATION: every code change bumps its SPARC document versions (Specification, Pseudocode, Architecture, Refinement, Completion) with a footer naming all five, and updates the tutorial, the pack reference, and the README in the same change.

  - ARCHIVE RULE: only the latest version of any versioned document lives in the working root; superseded versions move to archive/ (git mv) in the same change. I swept this root clean at close-out; keep it that way.

  - WORK-PACKAGE AND LAYER NUMBERING is a single global counter shared across the program; take the next free numbers, never reuse.

  - BRANCH-AND-PR: feature/ branch, pull request, squash-merge; no direct pushes to main. No Artificial-Intelligence attribution anywhere. No Roman numerals.

  - HOUSE FORMATTING for owner-facing plain-text documents: no markdown characters; bullets are "• " and nested "◦ "; a blank line between every sentence and between bullets; spell out an acronym on first use then use it in parentheses; define jargon on first use; use space-dash-space, never an em-dash; never draw lines across the page with dashes or equals signs; deliver as a downloadable plain-text artifact; web-search to ground claims before writing; explain at a novice level.
